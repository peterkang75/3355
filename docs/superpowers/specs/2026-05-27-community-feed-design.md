# 소식 커뮤니티 피드 — 설계 문서

> 작성일: 2026-05-27
> 상태: 설계 확정 (사용자 검토 중)
> 로드맵 위치: **Phase 9** (8-D/8-E보다 우선 — 사장님 결정 2026-05-27)

---

## 1. 배경 / 목적

지난 라운딩 사진으로 가는 동선이 약하다. 현재 경로는 `홈 → 라운딩 → '지난' 탭 → 갤러리`인데,
라운딩 페이지의 '지난' 토글이 중요도에 비해 눈에 안 띄고 입구가 작다.

**목적:** 사진을 중심으로 동호회 커뮤니티를 활성화한다. 회원들이 지난 라운딩 사진을
쉽게 보고, 반응(좋아요)·댓글로 소통하며, 자유롭게 일상 사진글도 올리게 만든다.

**방향 (사장님 확정):**
- 인스타그램식 **혼합 커뮤니티 피드** (라운딩 자동게시물 + 자유 사진글)
- 하단 네비 **전용 '소식' 탭**으로 진입 (탭 한 번)
- **반응(♡) + 댓글**까지 (알림은 v1 제외)

---

## 2. 역할 분리 (중복 입구 제거)

소식 탭이 생기면 라운딩 페이지의 '지난' 탭, 기존 공지게시판과 역할이 겹칠 수 있다.
세 화면을 **목적으로 분리**하여 중복을 없앤다:

| 화면 | 성격 | 보여주는 것 |
|------|------|-------------|
| **소식 탭** | 사진첩 + 소통 | 라운딩 사진 게시물 + 자유 사진글, 반응(♡), 댓글 |
| **라운딩 → '지난' 탭** | 기록/성적표 | 날짜, 참가자, 스코어, 시상(우승) 정보 — **사진 제외** |
| **공지게시판(`Post`, /board)** | 운영 공지 | 기존 그대로 유지, 손대지 않음 |

---

## 3. 네비게이션 구조

```
홈   라운딩   [소식]   참가비   더보기
                ▲ 신규 탭
```

- 하단 네비에 **'소식'** 탭 신설.
- 네비 항목 수 유지(5개)를 위해 운영자 전용 **'관리' 탭을 '더보기' 안으로 이동**.
  결과: 운영자/일반회원 모두 `홈·라운딩·소식·참가비·더보기` 5개.
- '소식' 탭 = 전체 화면 세로 스크롤 혼합 피드.

---

## 4. 피드 동작 (인스타식 혼합)

피드는 **두 종류의 게시물**이 **최신순으로 섞여** 노출된다.

### 4.1 라운딩 게시물 (자동 생성)

- **사진/영상이 1장 이상 있는 모든 지난 라운딩**이 자동으로 피드 게시물이 된다.
  - 타입 무관(정기모임·번개·컴페티션 등 전부).
- 별도 "발행" 단계 없음. 사진을 올리면 자동 등장.
- **제외:** 사진 0장, 미래(예정) 라운딩, 아카이브(`photosArchivedAt` 존재) 라운딩.

```
┌─────────────────────────────┐
│ 5월 정기모임 · 남촌CC · 5/18 │  ← 라운딩 정보
│ ┌─────────────────────────┐ │
│ │      대표 사진 (크게)    │ │  ← 첫 'ready' 사진, 탭 → 전체 갤러리
│ │                   [+8]  │ │     (기존 MediaGallery·업로드 재사용)
│ └─────────────────────────┘ │
│ ♡ 12      💬 5      📷 9장   │
│ OOO  날씨 최고였어요         │  ← 최근 댓글 1~2개 미리보기
│ [ 댓글 입력...            ]  │
└─────────────────────────────┘
```

- 대표 사진 = 첫 `status='ready'` 미디어(없으면 영상 썸네일). 탭 → 기존 `MediaGallery`로 전체 보기·업로드.

### 4.2 자유 사진글 (정회원이 직접 작성)

- **정회원**이 글(본문) + 사진(선택)을 자유롭게 작성. 라운딩 게시물 사이에 섞여 노출.
- 사진은 기존 R2 업로드 파이프라인 재사용.

```
┌─────────────────────────────┐
│ 👤 OOO · 2시간 전            │  ← 작성자 + 시각
│ 오늘 연습장 다녀왔어요 ⛳    │  ← 본문
│ ┌─────────────────────────┐ │
│ │        사진 (선택)       │ │
│ └─────────────────────────┘ │
│ ♡ 8       💬 3              │
│ [ 댓글 입력...            ]  │
└─────────────────────────────┘
```

- 작성 진입: 소식 탭의 "글쓰기(+)" 버튼.
- 본문/사진 중 최소 하나 필요.

### 4.3 정렬

- 라운딩 게시물·자유글을 **통합 타임스탬프** 기준 최신순.
  - 자유글: `createdAt`.
  - 라운딩 게시물: 최신 미디어 업로드 시각(새 사진 올라오면 위로). 단순화 시 라운딩 날짜로 대체 가능 — 구현 시 확정.

### 4.4 반응 (♡)

- **게시물 단위** 좋아요 토글(사진 1장 단위 아님). 카운트 + 내가 눌렀는지 표시.

### 4.5 댓글

- **게시물 단위** 댓글, 평면 구조(대댓글 없음).
- 본인 댓글: 수정·삭제. **운영자: 전체 삭제**.
- 카드엔 최근 1~2개 미리보기, 전체는 펼침/상세.

### 4.6 권한

| 구분 | 보기 | 반응(♡) | 댓글 | 자유글 작성 |
|------|:----:|:-------:|:----:|:-----------:|
| 정회원 | O | O | O | O |
| 게스트(`isGuest`) | O | O | **X** | **X** |

## 5. 데이터 저장 방식

기존 `Post`는 댓글·좋아요를 **JSON 컬럼**으로 저장하며, Plan.md **Phase 5-2**에
*"Post.comments JSON → Comment 모델 분리"* 가 숙제로 잡혀 있다. 피드 댓글을 JSON으로 만들면
향후 마이그레이션이 두 배가 되므로, **처음부터 제대로 된 공용 테이블**로 만든다.

### 신규 모델 (제안)

```
FeedPost                         # 자유 사진글
  id          String   @id
  authorId    String
  content     String?            # 본문(사진만 올릴 수도 → nullable, 단 본문/사진 중 1개 필수는 앱에서 검증)
  createdAt   DateTime
  updatedAt   DateTime
  @@index([authorId])

FeedPostMedia                    # 자유글 사진 (RoundingMedia와 동일 R2 방식)
  id          String   @id
  feedPostId  String
  type        String             # "photo" | "video"
  objectKey   String
  thumbnailKey String?
  ...(RoundingMedia와 동일 메타: fileSize, width, height, status 등)
  @@index([feedPostId])

Reaction                         # 공용(다형성): 라운딩 / 자유글 모두 수용
  id          String   @id
  targetType  String             # "booking" | "feedpost"
  targetId    String
  memberId    String
  type        String   @default("like")
  createdAt   DateTime
  @@unique([targetType, targetId, memberId, type])
  @@index([targetType, targetId])

Comment                          # 공용(다형성)
  id          String   @id
  targetType  String             # "booking" | "feedpost"
  targetId    String
  authorId    String
  content     String
  createdAt   DateTime
  updatedAt   DateTime
  @@index([targetType, targetId])
  @@index([authorId])
```

- `targetType`/`targetId` 다형성 → 나중에 공지(`Post`)도 같은 테이블로 흡수(Phase 5-2) 가능, 위험 없이 후속 처리.
- `createdAt` 보유 → 향후 알림 추가 시 스키마 변경 불필요.
- **v1 범위:** Booking(라운딩) + FeedPost(자유글) 대상만. 기존 `Post` JSON 마이그레이션은 후속(Phase 5-2)으로 분리.

## 6. 홈 배너 연동

- 기존 홈 "지난 정기모임" 자동 배너 유지하되, 클릭 시 **'소식' 탭으로 이동**(현재는 갤러리 직행).

---

## 7. 라운딩 '지난' 탭 재구성

- 기존 '지난' 탭의 **사진 썸네일/갤러리 진입 제거**.
- 지난 라운딩 **기록 카드**로: 날짜·장소·참가자·스코어 요약·시상(우승) 정보.
  - 스코어/시상 데이터는 기존 리더보드·스코어 로직 재사용.
- '예정'·'지난' 토글은 유지하되, '지난'은 사진 없는 정보 전용으로 성격 변경.

## 8. 엣지 케이스 (역검수)

- **사진 0장 라운딩:** 라운딩 게시물로 안 만들어짐.
- **아카이브된 라운딩(`photosArchivedAt`):** 피드 제외(원본 미디어 없음).
- **업로드 중/실패 미디어:** 대표 사진은 `status='ready'`인 것만. ready 0장이면 라운딩 게시물 제외.
- **자유글 본문·사진 둘 다 비었을 때:** 작성 불가(검증).
- **게시물 삭제 시:** Booking/FeedPost 삭제 → 연결된 Reaction/Comment·미디어도 정리(고아 레코드 방지).
- **게스트가 댓글/작성 시도:** UI 숨김 + 서버 거부(이중 방어).
- **자유글 작성자가 게스트로 강등/탈퇴:** 글 표시 정책 결정(작성자명 유지 권장).
- **빈 피드:** "아직 게시물이 없어요" 안내 + 첫 글/사진 유도.
- **성능:** 회원 수십 명 규모 → v1은 단순 최신순 로드(무한스크롤은 범위 외, 필요 시 "더 보기").

## 9. 안 만드는 것 (Out of Scope — v1)

- @멘션
- 사진 1장 단위 댓글/반응 (게시물 단위만)
- 푸시·텔레그램 알림 (테이블만 대비, 기능은 후속)
- 무한 스크롤 / 페이지네이션
- 대댓글(스레드)
- 기존 `Post`(공지게시판) JSON → 신규 테이블 마이그레이션 및 피드 통합 (Phase 5-2로 분리)
- 게스트의 댓글·자유글 작성

## 10. 구현 단위 (Step 분할 — 상세는 implementation plan에서)

1. DB: `FeedPost`·`FeedPostMedia`·`Reaction`·`Comment` 모델 추가 (db push)
2. API: 피드 목록(라운딩+자유글 통합) / 반응 토글 / 댓글 CRUD / 자유글 CRUD (권한 검증 포함)
3. 하단 네비 재구성: '소식' 탭 추가 + '관리' → '더보기' 이동
4. 소식 피드 페이지: 혼합 게시물 카드 목록 + 라운딩 게시물 대표사진 → 기존 갤러리 연결
5. 자유글 작성 화면(글+사진 업로드, 정회원)
6. 반응(♡) + 댓글 UI (공용 컴포넌트)
7. 홈 배너 → '소식' 연결
8. 라운딩 '지난' 탭 재구성: 사진 제거 + 정보/스코어/시상 표시

각 Step 완료마다 `npx vite build` 확인 후 다음 진행.
